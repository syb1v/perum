## 2. Критические проблемы

### 2.1 [SECURITY] Захват хоста платформы через slug школы
`SchoolCreate.slug` — голый `str` без паттерна и резерв-листа (`perum-core/app/routers/schools.py:42-46`),
в отличие от организации (`schemas/organization.py:7-45`: SLUG_PATTERN + RESERVED_SLUGS).
Slug школы напрямую формирует Caddy-хост (`school_provisioner.py:142`), маршрут вставляется в
**index 0 с terminal=true** (`caddy_admin.py:96-99`) — ДО статического блока `admin.{base}`.
**Любой org_admin может создать школу со slug=admin и перехватить `admin.avari-land.ru`** —
консоль и API платформы начнут проксироваться в контейнер его школы. Пробой границы platform↔org.
**Фикс:** общий SLUG_PATTERN + RESERVED_SLUGS (admin/api/www/docs/internal/platform) для школ;
в caddy_admin дополнительно запрещать host == admin.{base}.

### 2.2 Удаление организации оставляет работающие школы-«призраки»
`DELETE /api/organizations/{slug}` чистит ресурсы только по лейблу `com.perum.org=<org.slug>`
(`tenant_provisioner.py:258-276`), а школьные стеки помечены `com.perum.org=sch-<slug>`
(`stack_spec.py:57-60`, подтверждено на проде). При `purge=true` ORM-каскад удаляет строки School
из control-БД, но **docker-контейнеры школ продолжают обслуживать трафик при полном отсутствии
записи о них**. Дыра данных и изоляции.
**Фикс:** в deprovision(org) итерировать org.schools и вызывать deprovision_school для каждой.

### 2.3 Заморозка (freeze/suspend) отсутствует полностью
Нет ни статуса (`models.py:66-71,185-187`: только provisioning/active/failed/archived), ни
эндпоинтов, ни UI, ни механизма принуждения. Тенант не сверяет статус с ядром (исходящих вызовов
ноль), Caddy-маршрут и контейнер живут своей жизнью. `require_org_admin` не проверяет статус орг —
**org_admin даже архивированной организации сохраняет полный доступ** (`deps.py:37-55`).
**Фикс:** статус `suspended` у Organization/School + `POST /suspend|/unsuspend`
(platform_admin для орг, org_admin для школ) + реальное действие: docker stop app-контейнера
(том сохраняется) и/или подмена Caddy-маршрута на страницу «заморожено» + проверка статуса орг
в require_org_admin и в логине.

### 2.4 Удаление безвозвратно стирает тома без бэкапа
`remove_stack` удаляет контейнеры И тома (`docker_client.py:216-242`), pg_dump перед сносом нет,
grace-периода нет, UI org-консоли всегда шлёт `purge=true` (`org/page.tsx:102`). Одна кнопка +
браузерный confirm = все данные школы уничтожены без отката.
**Фикс:** автоматический pg_dump перед remove_stack; двухфазное удаление (archived с сохранением
тома N дней → purge).

### 2.5 [R4] Корневой домен ядра мёртв; лендинг — на поддоменах школ
- В `Caddyfile.prod` только блоки `admin.{base}` и `*.{base}` — апекса нет (`deploy/caddy/Caddyfile.prod:22,36`).
- `/internal/validate-domain` отклоняет апекс (`internal_domains.py:28`: только `endswith('.'+base)`) → сертификат не выпускается.
- Live: `https://avari-land.ru` → TLS alert internal error; `http://` → 308 на мёртвый https.
- Лендинг (`perum-web/src/app/page.tsx`) рендерится на `/` любого не-admin хоста (`middleware.ts:106-121`) — т.е. на корне каждой школы.
- SSR корня школы отдаёт только «Загрузка...» (`AuthContext.tsx:189`) — SEO лендинга нулевое.
- Форма лендинга `POST /api/contact` → **404 на всех хостах** (live), обработчика нет нигде; в тенанте секция «обращения» читает ContactInquiry, которые некому создавать.
- Неизвестный поддомен → 200 с тем же UI и `/api` → 503, страницы «школа не найдена» нет; on-demand TLS выпустил сертификаты на мусор (mail/www/vpn/случайные строки) — риск rate-limit Let's Encrypt.

**Фикс (пакетом):** апекс-блок в Caddyfile.prod (`/api/*` → perum_core, прочее → perum_web);
разрешить апекс в validate-domain; `isApexHostname()` в `lib/host.ts` + ветка в middleware
(лендинг ТОЛЬКО на апексе, вне AuthProvider); на школьных хостах `/` → redirect `/login`;
реализовать `POST /api/contact` в ядре (+ rate-limit) и раздел лидов в /platform;
ужесточить validate-domain до зарегистрированных School/SchoolDomain; кастомный not-found.tsx.

### 2.6 [R2] Биллинга нет
Весь биллинг: поле `organizations.plan` + словарь лимитов школ (`billing.py:10-15`,
trial=1/basic=5/pro=50/enterprise=1000) + 402 при создании школы сверх лимита. В 7 миграциях и
прод-БД (12 таблиц) — ни одной биллинговой. Провайдера нет (PLAN.md:351 — «пост-Phase-11»).
UI нет вообще: план нельзя ни увидеть, ни выбрать при создании, ни сменить (только curl).
Связь оплаты со статусом отсутствует как класс (нет paid_until/trial_ends_at, планировщика нет).
Обход лимита: возрождение archived-школы (reuse-ветка `schools.py:100-103`) и `reprovision`
лимит не проверяют. POST /api/organizations принимает любой plan без валидации (молча → лимит trial).
**Фикс-минимум (закрывает «управление»):** paid_until + статус suspended + авто-suspend при
просрочке (проверка на запросе, без планировщика) + экран биллинга в /platform + select плана при
создании + проверка лимита во всех трёх путях провижининга. Затем — ЮKassa.

### 2.7 [R3+R5] Статистики нет, телеметрии нет
- В ядре ноль stats-эндпоинтов (grep по роутерам → 0); единственные цифры — 6 Prometheus-гейджей из control-БД (`metrics.py:39-75`), без разреза по орг/школам, без продуктовых показателей.
- Телеметрия тенант→ядро не реализована: в perum-tenant **ноль исходящих HTTP-вызовов**, `CONTROL_PLANE_URL` — мёртвый конфиг. Обещанный доками heartbeat 30с (ARCHITECTURE.md:71) не существует.
- org_admin видит только метаданные стека (slug/status/release_tag); внутришкольная аналитика (KPI, успеваемость, посещаемость, deep-economy — `school_admin/service.py:148-190`) заперта за require_admin.
- Легаси-сегмент `/system-admin` в perum-web — труп: зовёт `/api/system/*`, которых нет ни в одном бэкенде. В легаси эта статистика существовала (`PERUM/app/routers/system.py:286,320,490`) — функциональность потеряна при портировании.
- Статусы школ в метриках — записи БД, не живое состояние: упавшая школа навсегда «active» (фоновых проб нет, app-контейнер создаётся без healthcheck — `docker_client.py:170-172` считает running здоровым).
- На проде Prometheus/Grafana **не запущены** (контейнеров нет, только тома), при этом фаза 9 «✅»; prometheus.yml не передаёт METRICS_TOKEN → при запуске получит 401; Grafana без единого дашборда.

**Фикс (один механизм закрывает R3 и стат-часть R5):** push-телеметрия из тенанта (фоновая
asyncio-задача, 60с, POST на CONTROL_PLANE_URL/api/telemetry с TELEMETRY_TOKEN — он уже
инжектится в env стека) → таблица снапшотов в control-БД → `GET /api/platform/stats`,
`/api/organizations/{slug}/stats` (platform_admin), `/api/schools/{id}/stats`, `/stats/summary`
(org_admin) → KPI-блоки в /platform и /platform/org → /metrics с лейблами org/school +
perum_school_up. Агрегаты без PII — инвариант изоляции не нарушается.

### 2.8 [R5] Управление админами школ не существует
Можно создать только первого school_admin при создании школы (bootstrap, временный пароль
показывается один раз — `schools.py:66-71`). Повторный bootstrap → 409, который провижинер молча
глотает (`school_provisioner.py:91-92`). Нет list/add/edit/delete/reset-password — ни в ядре, ни
в тенанте, ни в UI. **Потеря единственного админа = школа неуправляема навсегда.**
При этом CHANGELOG.md:19 и ROLES.md:56 объявляют `GET/POST/DELETE /api/admin/schools/{id}/admins`
существующими — это функциональность на бумаге.
**Фикс:** RPC ядро→тенант (расширить `/internal` операциями над User role=school_admin под
x-telemetry-token) + эндпоинты ядра `GET/POST/PATCH/DELETE /api/schools/{id}/admins` и
`POST .../reset-password` под require_org_admin + UI в org-консоли.

### 2.9 [R6] Геймификация неуправляема: маркет/квесты/биржа — заглушки
Админ-UI секций живой, бэкенда нет (live-подтверждено 404 на проде):
- Маркет: `MarketManagement.tsx` зовёт `/api/admin/market/*` — в тенанте 0 роутов; ShopItem создаётся только seed_test_data.py (на проде не запускается). Магазин в реальной школе вечно пуст.
- Квесты: `QuestManagement.tsx` зовёт GET/POST/PUT `/api/quests` — роутер содержит только студенческие take/claim/complete. Админ не может создать ни одного квеста.
- Биржа: `ExchangeManagement.tsx` зовёт `/api/exchange/admin/*` — нет; ExchangeSettings/TradingWindow только из seed.

Для геймифицированного журнала это ядро продукта. **Фикс:** реализовать admin-CRUD трёх модулей
под require_admin со school_id-скоупом (сервисный слой биржи уже скоупен); до реализации — скрыть
секции.

## 3. Высокие риски (high)

1. **Обход rate-limit логина через X-Forwarded-For** — и ядро, и тенант берут IP из первого
   элемента XFF, контролируемого клиентом (`perum-core/app/core/ratelimit.py:20-24`,
   `perum-tenant/app/core/ratelimit.py:20-22`; uvicorn `--forwarded-allow-ips=*`) → неограниченный
   перебор паролей. Фикс: request.client.host либо последний элемент XFF + лимит по login + Redis.
2. **docker.sock в perum_core + непроверяемый image релиза** — publish_release принимает любую
   строку image (`releases.py:20-24,76`), OTA пуллит и запускает на хосте. Компрометация
   platform_admin = root на хосте. Фикс: docker-socket-proxy, allow-list реестров (ghcr.io/syb1v),
   пиннинг по digest.
3. **sshd: root по паролю** (PermitRootLogin yes + PasswordAuthentication yes; fail2ban+ufw LIMIT
   смягчают). Фикс: ключи, PasswordAuthentication no, смена пароля (он уже фигурировал в переписке).
   Лишний sshd-порт 49122 (remnawave) убрать.
4. **platform_admin не может управлять школами** — весь /api/schools под require_org_admin
   (`main.py:137-142`); в /platform нет ни школ, ни счётчика. Решить: либо platform-эндпоинты
   школ (R1 буквально), либо официально закрепить делегирование в org_admin.
5. **Редактирования нет** — PATCH орг (name/admin_email/deployment_mode) и школ не существует;
   активную сущность нельзя переименовать (reuse-ветка обновляет только failed/archived).
6. **UI платформы пуст** — DELETE/reprovision/org-admins/enrollment-token/billing есть в API,
   но в /platform ни одной кнопки: оператор ядра управляет только curl'ом.
7. **Нет управления учётками org_admin** — только создание; деактивации/удаления/сброса пароля нет
   (is_active есть в модели, переключать нечем).
8. **Caddy-маршруты школ не переживают рестарт Caddy** — `_sync_caddy_routes` на старте ядра
   восстанавливает только легаси-таблицы Organization/OrganizationDomain, школьные маршруты
   восстанавливаются только если жив контейнер ядра с момента провижининга. После
   `docker restart caddy` (+ core) школы могут потерять маршрутизацию до reprovision.
9. **perum_web 10+ дней unhealthy** — healthcheck `wget localhost:3000` в busybox резолвит ::1,
   Next слушает только IPv4 → 30655 фейлов подряд; сервис при этом работает. Чинится healthcheck'ом
   на 127.0.0.1. Показательно: этого никто не заметил, потому что мониторинга нет.
10. **Диск прода 92%** (800 МБ свободно из 9.8 ГБ), RAM 2 ГБ + swap занят — провижининг следующей
    школы или docker pull может убить сервер.

## 4. Средние/низкие (сводно)

**Безопасность:** CORS ядра `allow_origins=['*']` + credentials (`main.py:99-105`);
`deploy/.env.prod` НЕ в .gitignore (git check-ignore подтвердил — риск коммита прод-секретов);
пароль control-БД захардкожен perum/perum (`docker-compose.core.yml:13,57`, CONTROL_DB_PASSWORD —
мёртвая переменная; любой контейнер сети perum_internal может читать control-БД);
validate-domain пропускает любой поддомен → абуз on-demand TLS (мусорные серты уже выпущены);
register-users не валидирует role (school_admin может плодить админов в обход org_admin —
`user_admin/router.py:40`, `service.py:182,218-222`); `_get_scoped` пропускает school_id=NULL
(`user_admin/service.py:50-54`); нет серверной ревокации сессий (JWT 7 дней, logout — no-op,
смена пароля не отзывает токены); мёртвая ветка org_admin в `_is_admin` журнала/учителя
(`journal/service.py:34`, `teacher/service.py:13`) — латентный риск; TELEMETRY_TOKEN сравнивается
не constant-time; bcrypt молча режет пароль до 72 байт; /api/agent/whoami и /health/db без auth.

**Прод-компоновка:** репозиторный compose публикует 2019/9090/3001 на 0.0.0.0 и монтирует
DEV-Caddyfile с неаутентифицированным admin — прод спасает только серверный оверрайд **вне VCS**
(docker-compose.prod.yml в репо отсутствует); app-контейнер школы без healthcheck → провижининг
и OTA «успешны» даже при упавшем приложении; deploy/scripts пуст (DEPLOYMENT.md ссылается на
несуществующий backup-org.sh), автобэкапов нет вообще; на сервере чужой софт (remnawave, warp-svc,
certbot b.ragenet.tech, кроны geodata/traffic-guard) — расширяет поверхность атаки.

**Биллинг:** org_admin не видит свой план/лимит (узнаёт по 402); понижение плана не сверяется
с использованием; TOCTOU-гонка проверки лимита; нет метрик по планам; тарифной оси кроме числа
школ нет (план в env школы не передаётся, лимитов учеников/хранилища нет).

**Тенант:** создание организации поднимает легаси org-стек `org_<slug>_*`, не нужный в v2
(на проде его и нет — поведение несогласованно); мёртвые элементы админки (clear-cache,
/admin/migration, /admin/system/schools); `update_user` не умеет менять role/is_active
(сменить роль = удалить с CASCADE всех оценок); внутришкольные FK не проверяются по школе/роли
(defense-in-depth); teacher_count=0 заглушка; online-users — честная заглушка;
/api/teacher/homework и /control-works возвращают [] при живом CRUD в coursework.

**Консистентность доков:** PROGRESS.md противоречит сам себе (строка 6: ≈78%, строка 23: ≈99%,
строка 207: версия 0.0.19 при CHANGELOG 0.0.32); CHANGELOG не ведётся с 0.0.32 (коммиты 49–74);
ROLES.md одновременно декларирует инвариант «org_admin не заходит внутрь школы» и примеры кода,
где org_admin = school_admin/teacher; TENANT_ISOLATION.md и DOMAINS.md целиком описывают старую
модель silo=организация без пометки «историческое»; строка «avari» не встречается в репозитории
вообще (всё зашито на perum.ru: host.ts, .env.prod.example, mailto contact@perum.ru на лендинге);
фаза 9 «✅» при незапущенной observability и нереализованной биллинг-валидации доменов;
X-School-Id удалён в 0.0.32, но память/доки местами на него ссылаются.

**Честная оценка готовности:** заявленные «≈99%» не соответствуют действительности относительно
требований владельца. По факту: изоляция и базовый lifecycle школ — крепкие; биллинг, статистика,
заморозка, лендинг, управление админами, геймификация-админка — отсутствуют или заглушки.
Реалистичная оценка готовности к коммерческой эксплуатации: **~70–75%**.

## 5. Что РАБОТАЕТ хорошо (подтверждено)

- **Изоляция школ** — трёхслойная и проверенная: стек на школу с отдельной БД, случайный

  последовательный school_id-скоупинг в service-слое; эскалация между школами/в организацию
  невозможна (R6-границы).
- **Прод-секреты** заданы корректно (SECRET_KEY 64 зн., не дефолт; BOOTSTRAP_ADMIN_PASSWORD не
  «admin»; SECRETS_ENCRYPTION_KEY работает — в БД `enc:`); Caddy admin API только на 127.0.0.1;
  /metrics защищён токеном и наружу не проксируется; /internal наружу не проксируется; fail2ban
  активен; все контейнеры restart=unless-stopped (переживают ребут).
- **OTA-обновление школ по кнопке** — volume-preserving, с автооткатом (соответствует
  принципу opt-in обновлений).
- Внутришкольная админка (предметы, классы, расписание, годы/периоды/звонки, журнал, PDF-импорт,
  ДЗ/КР, новости, апелляции, аналитика, пользователи) — реализована и school-scoped.
