# Версии (счётчик по коммитам)

> Версия = порядковый номер коммита `№(N)`. Здесь — журнал всех коммитов с датой
> и временем фиксации. Человекочитаемый журнал изменений по релизам `0.0.x` —
> в [CHANGELOG.md](../CHANGELOG.md). Правило: каждый новый коммит добавляет
> строку сюда — `№(N) · ГГГГ-ММ-ДД ЧЧ:ММ · хеш · описание`.

## Обновление 2026-06-13 (версионирование релизов)

Зафиксирована модель версионирования образов и релизов тенанта (детали —
в [RELEASING.md](RELEASING.md), здесь не дублируются):

- **Контентная привязка образов.** Каждый образ (`perum-core`/`perum-tenant`/
  `perum-web`) тегается по коммиту — `git-<sha>` (+ `latest`). Тег `git-<sha>`
  однозначно привязан к содержимому, поэтому одинаковый код = одинаковый тег.
  CI (`release.yml`) по paths-filter собирает и пушит в GHCR **только изменённые**
  образы — компоненты обновляются независимо.
- **Релиз тенанта = `source_commit` + интегрити-гард.** Релиз школы привязан к
  реальному коду через `Release.source_commit` (миграция control-БД `0014`).
  Публикация **отклоняет no-op**: релиз, чей образ/коммит совпадает с текущим
  активным, не регистрируется (нет «пустого» обновления). CI авто-регистрирует
  релиз тенанта (`POST /api/ci/release` по `RELEASE_PUBLISH_TOKEN`).
- **Ченджлоги — авто из `git log`.** CI генерирует changelog из истории git;
  он виден в таблице релизов консоли ядра (версия/образ/коммит/changelog) и в
  баннере «Доступно обновление + changelog» в портале орг.
- **Архитектура v2 (silo = ШКОЛА).** Каждая ШКОЛА — отдельный docker-стек
  (app + БД + volume); `School` — ребёнок `Organization` (`School.org_id`).
  Провижинит/обновляет школы узел орг (`org_admin`); ядро держит только метаданные.
  Провижининг/реправижининг/обновление школы — асинхронные (`202` + фоновая
  задача под school-локом, `app/core/locks.py`). Ядро **не** монтирует
  `docker.sock`: к демону ходит через `docker_proxy` (RO, фильтр API).
- **Изоляция токенов.** `internal_rpc_token` разведён с `telemetry_token`
  (миграция `0013`): тенант с заданным `INTERNAL_RPC_TOKEN` принимает на
  `/internal` только его (constant-time сравнение).

Аудит иерархии ядро→организация→школа пройден, находки закрыты —
см. [AUDIT_2026-06-12.md](AUDIT_2026-06-12.md).

## Готовность продукта (оценка на №82)

**≈ 99 %** по работающим фичам (взвешенно). Сверх фаз закрыты бэклог hardening
(шифрование секретов at-rest, rate-limit логина, защита `/metrics`, вложения на
volume) и продуктовые хвосты (настройки школы, уведомления, обращения).
**Фазы 4–10 закрыты, Фаза 11 на ~75%**
(CI pytest+tsc, release-пайплайн образов в GHCR, прод-env/Caddyfile/RUNBOOK готовы;
остаётся выкат на реальный сервер с доменом/реестром — вне локальной среды). Фаза 10:
RBAC-матрица + тесты ядра (56 passed), isolation E2E (cross-school/cross-level 401,
отдельные БД), k6-нагрузка (p95 ~2.6ms, 0% ошибок). Фаза 4: кастомные
домены школ + on-demand TLS gate (`/internal/validate-domain`). Фаза 9: биллинг-
заглушки (план + лимит школ), observability (`/metrics` + Prometheus + Grafana) и
обновления «по кнопке» (OTA). Плюс выполнена
**переархитектура v2 «узел организации»** (silo = ШКОЛА): ядро провижинит изолированные
школьные стеки (контейнер+БД на школу); `org_admin` управляет школами и **обновляет их
«по кнопке» (OTA, с откатом)** через портал орг; платформа публикует релизы; агент орг
вынесен (enroll-on-boot); тенант = одна школа с `school_admin`. Это закрывает ключевое
требование «обновление по воздуху по кнопке». Дальше — финальный выкат Фазы 11 на
реальный сервер (домен + GHCR + secrets, по RUNBOOK) и полный мульти-сервер (школы в
БД агента). Нюанс: фронтенд
(легаси, ~47k LOC) **скопирован целиком**.

## Темп и сроки (оценка)

- Факт: ~44% за ~4 ч фокус-работы (07:54→11:51, 19 коммитов). Это **не календарные
  недели из PLAN.md (16–20 нед.)** — в режиме AI-пары темп на 1–2 порядка быстрее.
- Прогноз (грубо, при том же темпе, «готово» = работает в dev, без полного QA):
  - до **полнофункционального продукта** (через Фазу 8): **~5–7 ч** фокус-работы.
  - до **прод-готовности** (Фазы 9–11: hardening, observability, деплой): **+3–5 ч**.
  - итого с текущего момента: **~8–12 ч** фокус-сессий.
- Календарь зависит от частоты сессий и времени на проверку/решения с твоей
  стороны. Сложная Фаза 7 (биржа/маркет) может идти дольше оценки.
