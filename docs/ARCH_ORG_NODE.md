# Архитектура v2 — «узел организации» (school = стек, OTA-обновления)

> **Статус:** принятое целевое направление (решение от 2026-05-25). Заменяет
> прежнюю модель «silo = организация, школы — строки в одной БД». Прежний дизайн
> описан в [ARCHITECTURE.md](ARCHITECTURE.md); этот документ — целевой.
>
> **Зачем:** требования владельца — физическая изоляция школ (контейнер на школу),
> минимум настройки на новом сервере / развёртывание из коробки, обновление «по
> воздуху» без физического доступа к серверу, обновление организатором по кнопке
> без ручного вмешательства. Один сервер сейчас → отдельные серверы под орг
> потом, **без переписывания**.

## Топология

```
ГЛАВНЫЙ СЕРВЕР — ЯДРО (perum-core)            ← platform_admin
  • реестр организаций
  • КАНАЛ РЕЛИЗОВ: версия образа (тег) + changelog
  • выдаёт enrollment-токены, координирует домены/TLS
  • НЕ имеет доступа к Docker серверов орг — только HTTPS-обмен (статусы, релизы)

СЕРВЕР ОРГАНИЗАЦИИ — УЗЕЛ ОРГ (org node)      ← org_admin (скоуп = своя орг)
  (свой сервер на орг ИЛИ тот же хост, что и ядро — один и тот же код)
  • «агент орг» (бывший tenant_provisioner) + Caddy + малая control-БД орг
  • портал орг: org_admin провижинит/останавливает школы и заводит их админов
  • провижинит ШКОЛЬНЫЕ СТЕКИ локально на своём Docker:
        school_<slug>_app + school_<slug>_db + volume   (контейнер на школу)

СТЕК ШКОЛЫ (school stack)                     ← school_admin/director/teacher/student/parent
  • это нынешний perum-tenant, «схлопнутый» до ОДНОЙ школы
  • журнал, оценки, маркет, биржа, квесты, новости, аналитика, апелляции
  • полностью изолирован: своя БД, свой volume, свой контейнер
```

## Роли по уровням

| Роль | Где живёт | Что делает |
|---|---|---|
| `platform_admin` | ядро | CRUD организаций, публикация релизов, enrollment, мониторинг |
| `org_admin` | узел орг (control-БД орг) | провижинит/останавливает школы своей орг, заводит/снимает админов школ, жмёт «обновить». **Внутрь школы не заходит.** |
| `school_admin`/`director` | стек школы | полный админ одной школы, изолирован |
| `teacher`/`student`/`parent` | стек школы | как сейчас |

`org_admin` относится к школам ровно как ядро к организациям (см. [ROLES.md](ROLES.md)).

## Потоки

### Enrollment (новый сервер орг — из коробки)
1. На новом сервере: установить Docker + запустить **один** bootstrap-`compose`
   (агент орг + Caddy + control-БД орг) с `ENROLLMENT_TOKEN` (выдаёт ядро).
2. Агент при старте регистрируется в ядре по токену (HTTPS), получает свой
   `org_slug`, секреты, текущий тег релиза.
3. Поднимает портал орг. Всё — ручной настройки больше нет.

### Провижининг школы (org_admin)
1. org_admin в портале орг: «создать школу» (имя, slug).
2. Агент локально поднимает `school_<slug>_app/_db/volume` (реюз нынешнего
   провижинера), прописывает маршрут в Caddy (`<slug>.<org-домен>`), прогоняет
   миграции, бутстрапит `school_admin` (одноразовый пароль возвращается org_admin).
3. Школа изолирована: своя БД/volume/контейнер.

### Обновление «по воздуху» по кнопке (опт-ин)
1. Ядро публикует релиз: новый тег образа + changelog в канал релизов.
2. Портал орг сравнивает свой текущий тег с каналом → баннер «доступно vX.Y».
3. org_admin жмёт «Обновить» → агент орг сам делает `docker pull` нового образа
   и **пересоздаёт** контейнеры (школ/портала) с сохранением volume; Alembic-
   миграции на старте. **Без SSH, без физического доступа.**
4. Принуждения нет — даже секьюрити-патчи только по кнопке (см.
   [update-model](../README.md) и память проекта). Откат — на предыдущий тег.

## Гарантии изоляции
- Школы: разные контейнеры + разные БД + разные volume (физ. + лог.).
- Орг: разные узлы (свой сервер на орг) либо как минимум разные стеки на хосте.
- Ядро не имеет доступа к данным внутри школ/орг — только метаданные/релизы.

## Что меняется относительно текущего кода
- **perum-core:** + сущность `School` (ребёнок `Organization`), + канал релизов
  (`Release`), + enrollment-токены, + auth `org_admin` уровня ядра/узла.
  Текущий `tenant_provisioner`/`stack_spec`/`docker_client`/`caddy_admin`
  переиспользуются как «агент орг», нацеленный на **школьные** стеки.
- **perum-tenant → стек школы:** `Organization`/мульти-`School` схлопываются в одну
  `School`-мету; bootstrap создаёт `school_admin`; из тенанта уходят `org_admin`,
  schools-CRUD, `OrgConsole`, остатки `X-School-Id`.
- **perum-web:** портал платформы (как есть) + портал орг (org_admin: школы+админы,
  +кнопка обновления) + школьное приложение на поддомене школы.

## Поэтапный план

1. ✅ **Ядро — данные и релизы:** модели `School`, `SchoolSecret`, `SchoolDomain`,
   `Release` (миграция `0003`). Аддитивно, текущее провижининг орг не затронуто.
2. ✅ **Агент орг — провижининг школ:** `school_provisioner` +
   `build_school_stack_spec` (реюз docker/caddy, namespaced label `sch-<slug>`);
   модель `OrgAdmin` + auth ядра (role+org_id) + `require_org_admin` (миграция
   `0004`); роутер `/api/schools` (CRUD/provision/reprovision/delete, scope по
   org_id); `platform_admin` заводит org_admin; bootstrap админа школы. **E2E ок.**
3. ✅ **Тенант → одна школа:** bootstrap создаёт `school_admin` (привязан к
   единственной школе стека) вместо org_admin; из тенанта убраны org-уровневые
   schools-CRUD/admins, `service_schools`, `require_org_admin`, фронт `OrgConsole`/
   `SchoolManagement` и ветка org_admin. **E2E ок** (новый стек → school_admin;
   `/api/admin/schools` в стеке 404). Схему БД оставили (одна школа на стек).
4. ✅ **OTA-обновления:** канал релизов (`/api/releases`, publish — platform_admin);
   `org_admin` видит `update-status` и жмёт `/api/schools/{id}/update` → агент
   свапает только app-контейнер на новый образ (БД/том целы) + миграции; при сбое —
   автоматический откат на прежний образ. **E2E ок** (v1→v2 с сохранением данных;
   битый релиз → rollback). Без доступа к серверу, опт-ин.
5. 🔶 **Enrollment + агент:** выдача одноразового токена (`POST /api/organizations/
   {slug}/enrollment-token`) + публичный handshake `POST /api/enroll` +
   bootstrap-шаблон `deploy/org-node/`. **Агент вынесен**: тот же образ ядра в
   `ROLE=org_agent` на старте сам подключается к ядру (enroll-on-boot), хранит
   идентичность локально (`agent_state`), `GET /api/agent/whoami`. **E2E ок**
   (отдельный контейнер-агент → enrolled=acme). Остаётся для полного мульти-сервера:
   перенос владения школами в БД агента + роутинг портала орг на агент.
6. ✅ **Фронт:** портал орг (`/platform/org`: консоль школ — провижининг,
   `update-status` + кнопка «Обновить» (OTA), удаление); у платформы — секция
   «Релизы» (публикация); ролевой роутинг по JWT; школы — на своих поддоменах.
7. ✅ **Доки/Caddy/деплой:** `ROLES.md` приведён к v2-иерархии (платформа → узел орг
   → стек школы); `ARCHITECTURE/PROVISIONING/DEPLOYMENT` помечены ссылкой на v2;
   `docs/RUNBOOK.md` (эксплуатация); прод-`deploy/caddy/Caddyfile.prod` (auto-HTTPS
   + on-demand TLS); `deploy/org-node/` (узел из коробки).

**Сделано:** этапы 1–4, 6 (полностью) + 5 (токен/handshake/bootstrap-шаблон;
остаётся вынос агента). Ядро провижинит школьные стеки; `org_admin` управляет
школами и обновляет их «по кнопке» через портал орг; платформа публикует релизы;
тенант = одна школа с `school_admin`; есть enrollment-токены + шаблон узла.
Агент вынесен (ROLE=org_agent + enroll-on-boot); доки/Caddy/деплой под v2 готовы
(этап 7). **Все 7 этапов v2 закрыты** (этап 5 — без полного мульти-сервера).
Осталось для полного мульти-сервера: владение школами в БД агента + роутинг
портала орг на агент (на одном хосте не требуется).

## Обновление 2026-06-13

- **docker_proxy вместо сырого сокета (#7).** Ядро/узел **больше не монтируют**
  `/var/run/docker.sock`. Доступ к демону идёт через сервис `docker_proxy`
  (`tecnativa/docker-socket-proxy`): сокет смонтирован к нему read-only, прокси
  фильтрует Docker API, ядро/агент обращаются по `DOCKER_HOST=tcp://docker_proxy:2375`.
  Полный вынос провижининга в отдельный `org-agent` (связка `proxy → agent` по
  HTTP) — следующий этап мульти-сервера, ещё впереди.
- **Разведение токенов на `/internal` (#6).** `internal_rpc_token` отделён от
  `telemetry_token` (миграция control-БД `0013`): ядро шлёт оба заголовка, а тенант
  с заданным `INTERNAL_RPC_TOKEN` принимает на `/internal` **только** его (telemetry-
  токен туда больше не пускают); сравнение constant-time.
- Аудит иерархии ядро → орг → школа пройден, находки закрыты — см.
  [AUDIT_2026-06-12.md](AUDIT_2026-06-12.md). Релизный процесс и CI описаны в
  [RELEASING.md](RELEASING.md) (не дублируется здесь).

## Открытые вопросы / риски
- Реестр образов: свой (Harbor) или публичный с приватным каналом тегов.
- Доставка релиза узлу: pull по кнопке (выбрано) vs webhook. Pull проще и
  безопаснее (узел инициирует, ядро не лезет внутрь).
- Безопасность Docker-доступа: ядро/узел больше **не монтируют сырой
  `/var/run/docker.sock`**. Сокет (RO) видит только сервис `docker_proxy`
  (`tecnativa/docker-socket-proxy`, фильтр Docker API), а ядро/агент ходят к демону
  по `DOCKER_HOST=tcp://docker_proxy:2375`. Полный вынос провижининга в отдельный
  `org-agent` (proxy → agent по HTTP) остаётся будущим этапом мульти-сервера.
- Миграции при обновлении: только вперёд; backup volume перед recreate.
- Это **самая крупная переделка**, переписывает Фазы 1–3 и сливается с «обновления
  по кнопке» из Фазы 9. Многоэтапно.
