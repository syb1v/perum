# Воркер ноды (org_agent) — как он работает

> **TL;DR.** Воркер — это тот же образ `perum-core`, запущенный в режиме
> `ROLE=org_agent` на сервере-ноде. Он подключает («enroll») ноду к организации,
> принимает команды от ядра по HTTP (`/api/agent/*`) и разворачивает/обслуживает
> школьные стеки **прямо на ноде**, а ядро остаётся единым мозгом и витриной.
> Это прослойка между **ядром** (control plane), **школами** (tenant-стеки) и
> **инфраструктурой** (серверы организаций).

---

## 1. Зачем воркер существует

Архитектура ПЭРУМ — silo-per-SCHOOL: каждая школа это изолированный Docker-стек
(контейнер приложения + своя БД + том). Эти стеки должны где-то крутиться. Есть
два варианта размещения:

| Размещение | Где крутится школа | Кто разворачивает |
|---|---|---|
| **Хост платформы** | на том же сервере, что и ядро | ядро напрямую (локальный Docker) |
| **Нода организации** | на отдельном сервере орг | **воркер** по команде ядра |

Воркер закрывает второй вариант: он даёт организации горизонтальное
масштабирование (добавил сервер → добавил ёмкость под школы), не отдавая ей
управления — все решения принимает ядро.

```
            ┌──────────────────────────────────────────────┐
            │                   ЯДРО (platform)             │
            │  org_admin UI · планировщик · биллинг · релизы │
            └───────────┬───────────────────────┬───────────┘
                        │ HTTP /api/agent/*      │ маршрут <slug>.<base>
                        │ (команды + метрики)    │ (reverse-proxy на ноду:80)
              ┌─────────▼─────────┐    ┌──────────▼──────────┐
              │   ВОРКЕР (node 1) │    │   ВОРКЕР (node 2)   │
              │  школы орг A      │    │   школы орг B       │
              └───────────────────┘    └─────────────────────┘
```

## 2. Что внутри ноды

Bootstrap-скрипт (раздел «Инфраструктура» → «Создать ноду» → «Скачать скрипт
установки») поднимает на сервере такой стек (`docker compose`):

| Контейнер | Образ | Роль |
|---|---|---|
| `perum_agent` | `ghcr.io/.../perum-core:latest` | **воркер** (`ROLE=org_agent`), API на `:3001`→`:3000` |
| `perum_node_db` | `postgres:15` | локальная БД ноды (`agent_state`, локальные записи школ/секретов) |
| `shared_redis` | `redis:7` | redis для школьных стеков ноды |
| `docker_proxy` | `tecnativa/docker-socket-proxy` | фильтрованный доступ к Docker-демону (воркер не монтирует сырой сокет) |
| `caddy` | `caddy:2` | локальный роутер: по `Host` отдаёт на контейнер нужной школы |

Плюс при работе на ноде появляются стеки школ: `school_<slug>_app`,
`school_<slug>_db` + тома `school_<slug>_data`, `school_<slug>_appdata`.

## 3. Подключение ноды (enroll-on-boot)

```
1. platform_admin: «Создать ноду» → ядро выдаёт одноразовый ENROLLMENT_TOKEN
   (привязан к организации) и кладёт его в bootstrap-скрипт.
2. На сервере: bash perum-node-*.sh → ставит Docker, поднимает стек.
3. Воркер при старте (enroll_on_boot):
     POST {CORE_URL}/api/enroll  { token, cpu_cores, ram_gb, disk_gb, agent_version }
4. Ядро валидирует токен → привязывает Node к организации, ставит status=active,
   снимает реальное железо, возвращает { org_slug, current_release }.
5. Воркер сохраняет это в локальную таблицу agent_state. Готово — нода в строю.
```

Идемпотентно: если нода уже подключена (`agent_state` есть) — повторный enroll не
делается. Токен одноразовый и живёт 7 дней.

---

## 4. Связь ядро ↔ воркер

Всё общение — обычный REST. Воркер слушает на порту `AGENT_PORT` (по умолчанию
`3001`), роуты под префиксом `/api/agent`. Каждый запрос ядра несёт
`Authorization: Bearer <AGENT_TOKEN>` (общий секрет, задаётся одинаково в ядре и в
compose ноды). Маршрутизация и аутентификация — `app/services/remote_node_client.py`
(сторона ядра) и `app/agent/router.py` (сторона воркера).

### Эндпоинты воркера

| Метод + путь | Назначение | Авторизация |
|---|---|---|
| `GET /api/agent/whoami` | liveness + кто я (для ping/мониторинга) | открыт |
| `GET /api/agent/health` | реальные метрики (CPU/RAM/диск/аптайм, psutil) | открыт |
| `GET /api/agent/schools` | какие школьные контейнеры крутятся | открыт |
| `POST /api/agent/schools/provision` | развернуть школу на ноде | `AGENT_TOKEN` |
| `POST /api/agent/schools/{slug}/update` | OTA-обновление школы | `AGENT_TOKEN` |
| `POST /api/agent/schools/{slug}/suspend` | заморозить школу | `AGENT_TOKEN` |
| `POST /api/agent/schools/{slug}/unsuspend` | разморозить | `AGENT_TOKEN` |
| `POST /api/agent/schools/{slug}/deprovision` | архивировать/удалить | `AGENT_TOKEN` |
| `POST /api/agent/schools/{slug}/internal-rpc` | проксировать внутренний RPC школы (управление админами) | `AGENT_TOKEN` |
| `POST /api/agent/restart` | полная перезагрузка стека ноды | `AGENT_TOKEN` |

**Важно: воркер — приёмник, а не инициатор** жизненного цикла. Он не опрашивает
ядро на предмет «что сделать» — ядро само шлёт ему команды, когда org_admin
нажимает кнопку. Единственное, что воркер инициирует сам — enroll при старте.

## 5. Воркер как прослойка между разделами проекта

| Раздел проекта | Как через него проходит воркер |
|---|---|
| **org_admin UI** | org_admin жмёт «Создать/Обновить/Заморозить школу» → ядро → воркер выполняет на ноде. Org_admin не знает, что школа на ноде — для него это одна кнопка. |
| **Планировщик нод** | при создании школы `NodePlanner.find_best_node` выбирает ноду орг (с учётом лимита оператора и ресурсной оценки) → создаётся `NodeAssignment` → ядро шлёт `provision` воркеру. |
| **Релизы / OTA** | текущий релиз (`Release.image`) ядро передаёт воркеру в команде; воркер pull'ит образ из реестра и пересоздаёт `school_*_app`. Версии — `x.y.z` из `perum-tenant/VERSION`. |
| **Маршрутизация / домены** | школа на ноде доступна по `<slug>.<base>`: wildcard DNS → платформа (TLS) → reverse-proxy на `нода:80` → Caddy ноды по `Host` → контейнер школы. DNS менять не надо. |
| **Управление админами школы** | контейнер школы в сети ноды недоступен ядру напрямую → ядро шлёт `internal-rpc` воркеру, тот ходит в `/internal` школы локально. |
| **Мониторинг** | монитор-петля ядра раз в `NODE_MONITOR_INTERVAL_S` пингует `whoami` (латентность) и тянет `health` (CPU/RAM/диск) → пишет снимок в `nodes.last_*`. UI рисует живые бары. |
| **Биллинг / лимиты** | лимит школ на орг и `node.max_schools` проверяются в ядре ДО отправки команды воркеру. |

## 6. Жизненный цикл школы на ноде — примеры

### Создать школу (полностью автоматически)

```
org_admin → POST /api/schools { slug, name, admin_email }
  ядро: find_best_node(org) → node 5 → NodeAssignment(school, node5)
  ядро: собирает секреты школы + текущий образ релиза
  ядро → POST нода5/api/agent/schools/provision { slug, name, image, db_password, ... }
    воркер: создаёт сеть → БД школы → app школы → миграции → сид → админ
    воркер: добавляет маршрут в Caddy ноды (Host=<slug>.<base> → school_app:3000)
  ядро: добавляет маршрут на платформе (<slug>.<base> → нода5:80)
  ядро: school.status=active, release_tag=image
→ школа сразу доступна: https://<slug>.<base>
```

Org_admin не делает ни одного ручного шага после «Создать школу».

### Обновить школу

```
org_admin → POST /api/schools/{id}/update
  ядро → POST нода/api/agent/schools/{slug}/update { image }
    воркер: pull нового образа → пересоздаёт app (том БД цел) → миграции
            при сбое — авто-откат на прежний образ
  ядро: запись в update_history (success | rolled_back | failed)
```

### Заморозить / разморозить / удалить

```
suspend   → воркер стопает контейнеры школы (том сохранён), ядро → status=suspended
unsuspend → воркер поднимает контейнеры, ядро → status=active
delete    → воркер: бэкап БД (+ вложений) → снос стека (archive: тома целы /
            purge: тома и запись удаляются), ядро снимает маршрут платформы
```

## 7. Перезагрузка стека ноды

Кнопка «перезагрузить» у online-ноды делает **полную** перезагрузку стека ноды,
включая сам воркер:

```
ядро → POST нода/api/agent/restart
  воркер: рестартует все контейнеры compose-проекта КРОМЕ себя (БД, redis, caddy,
          docker_proxy, школы), отвечает ядру
  воркер: через ~1.5с (ответ уже ушёл) рестартует САМ СЕБЯ в фоне
  ядро: сразу помечает ноду offline (связь рвётся)
  монитор ядра: через 5-15с воркер поднялся → ping ок → status снова active
```

То есть связь ядро→воркер на время пересборки **пропадает и возвращается** — это
ожидаемое поведение, в UI видно переход online → offline → online. Сервер при этом
не перезагружается, только Docker-контейнеры.

## 8. Мониторинг ноды (реальное время)

Монитор-петля ядра (`_node_monitor_loop`, `app/main.py`) по каждой ноде:

1. **Латентность + liveness** — лёгкий `GET whoami`, замеряет `ping_ms`.
   Недоступна → `status=offline`. Доступна → `status=active` + свежий heartbeat.
2. **Загрузка** — `GET health` воркера (psutil): `cpu_percent`, `ram_used/total`,
   `disk_used/total` → снимок в `nodes.last_*` + `metrics_at`.

UI раздела «Инфраструктура» поллит ноды каждые 10с и рисует живые бары CPU / ОЗУ /
ПЗУ / школы + чип латентности. Статус-иконка/пилюля берутся из `node.status`
(источник истины — монитор ядра), фронт его не пересчитывает.

> **Ёмкость по школам.** Знаменатель шкалы «X / Y школ» = `node.max_schools` —
> лимит, который задал оператор. Отдельно считается ресурсная оценка
> (`recommended_max`, по CPU/RAM ≈192 МБ и 0.15 ядра на школу). Планировщик не
> назначает сверх `min(лимит, оценка)`, чтобы не перегрузить железо.

## 9. Конфигурация воркера (ENV на ноде)

| Переменная | Назначение |
|---|---|
| `ROLE=org_agent` | включает режим воркера |
| `ENROLLMENT_TOKEN` | одноразовый токен подключения к орг |
| `AGENT_TOKEN` | секрет аутентификации команд ядра |
| `CORE_URL` / `CONTROL_PLANE_URL` | публичный URL ядра (enroll, телеметрия школ) |
| `DATABASE_URL` | локальная БД ноды (`perum_node`) |
| `DOCKER_HOST=tcp://docker_proxy:2375` | доступ к Docker через фильтрующий прокси |
| `CADDY_ADMIN_URL=http://caddy:2019` | управление маршрутами Caddy ноды |
| `SHARED_REDIS_URL` | redis ноды для школьных стеков |
| `IMAGE_REGISTRY` | реестр базовых образов (в РФ — `mirror.gcr.io`) |
| `TENANT_IMAGE` | образ школы по умолчанию (текущий релиз) |
| `PUBLIC_BASE_DOMAIN` | базовый домен для `<slug>.<base>` |

Соответствующие настройки на стороне ядра: `AGENT_TOKEN`, `AGENT_PORT`,
`AGENT_IMAGE`, `PUBLIC_CORE_URL` (см. `app/core/config.py`).

## 10. Безопасность и ограничения

- **Аутентификация команд** — `AGENT_TOKEN` (Bearer) на всех мутирующих эндпоинтах;
  `whoami`/`health` открыты как liveness.
- **Docker-доступ** — только через `docker_socket_proxy` (контейнеры/образы/тома/
  сети/exec), сырой сокет в воркер не монтируется.
- **Изоляция школ** — каждый стек имеет свои секреты (`db_password`, `secret_key`,
  `telemetry_token`, `internal_rpc_token`), которые генерирует ЯДРО и передаёт
  воркеру (совпадают с записью ядра — важно для бэкапов/управления).

### Известные ограничения (follow-up)

- **Транспорт ядро↔воркер по HTTP** (не TLS) — защищён `AGENT_TOKEN`, но для
  публичной сети желателен TLS/mTLS.
- **Секреты на ноде в открытом виде**, если не задан `SECRETS_ENCRYPTION_KEY`.
- **Purge вложений на ноде** — если бэкап тома вложений не снимается, удаление
  деградирует до архивации (тома сохраняются). См. CHANGELOG.
